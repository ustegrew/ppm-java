23 /**
24  * An atomic-access wrapper around a <code>FloatBuffer</code>. Designed
25  * for situations where we need to transport data between a high priority
26  * thread and a low priority thread in a producer-consumer scenario.
27  * Example scenario: A thread operating audio hardware sends audio sample data
28  * to another thread updating the GUI. The audio hardware thread cannot
29  * afford to be blocked, hence has high priority. The GUI thread isn't time
30  * critical, and we can afford to have it loose some data.
31  * <p><br/></p>
32  *
33  * The <code>TAtomicBuffer</code> guarantees that data can be passed from (to)
34  * a high priority thread within a fixed (short) time without
35  * blocking it. This comes at a cost - the associated low priority thread
36  * does not have the guarantee that all data actually reaches it - in highly
37  * contended situations the <code>TAtomicBuffer</code> would simply drop
38  * data where the low priority thread cannot keep up. If the low priority
39  * thread can't keep up with the streaming data it can't make full use of
40  * the stream anyway, so this seemed to be the best solution for data in
41  * contention situations.
42  * <p><br/></p>
43  *
44  * The <code>TAtomicBuffer</code> copies the incoming data frames before
45  * they are passed on to the consumer. This is to protect the producer.
46  * If we pass on uncopied data frames and the consumer side does changes
47  * to the data then those changes would be seen on the producer side as
48  * well and cause unforeseen side effects and hard to track down errors.
49  * The extra protection comes with a performance penalty as the copying
50  * costs time. To protect the time critical thread from the performance loss
51  * we can set the time point at which the data is copied. If the producer
52  * is the time critical we copy when the consumer fetches. If the consumer is
53  * time critical we copy when the producer sets it. In some situations
54  * the data copy may be unnecessary in which case we
55  * can instruct the atomic buffer to omit any data copying. However, due
56  * to the real possibility of data corruption the data must then be copied
57  * separately, either on the producer or on the consumer side. The copy
58  * actions can be set at construction time via the <code>copyPolicy</code>
59  * parameter.
60  *
61  * To monitor the quality of the data flow we offer various counters which can
62  * be queried by all parties (all {@link #ClearStats() clearable}}:
63  * <p><br/></p>
64  *
65  * <ul>
66  * <li>Overrun counter: Increments each time the producer pushes a new
67  * packet whilst an old packet still awaits collection
68  * by the consumer. A steady increase of the overruns
69  * indicates that the consumer can't keep up with the
70  * producer and looses packets.</li>
71  * <li>Underrun counter: Increments each time the consumer requests a packet
72  * whilst none has been pushed since the last collection.
73  * A steady increase of underruns indicates that the
74  * producer can't keep up with the consumer.</li>
75  * <li>Contention counter: Increments each time the producer and the consumer are
76  * both trying to acquire access to the buffer at the
77  * same time. A steady increase of the contentions
78  * indicates that both, producer and consumer, are too
79  * aggressive trying to set and get data.</li>
80  * </ul>
81  * <p><br/></p>
82  *
83  * Clients can use the underrun and overrun counters for automatic dropout compensation.
84  * For example, a low priority consumer thread can, with each cycle:
85  * <ul>
86  * <li>read overrun and underrun counts</li>
87  * <li>compute the difference: <code>d = nOver - nUnder</code>. If
88  * <code>d</code> is zero: All is well. If it's negative: Underruns.
89  * If it's positive: Overruns.</li>
90  * <li>then compensate by adjusting the time to the next cycle:</li>
91  * <li>
92  * <ul>
93  * <li><code>d == 0</code>: No adjustment.</li>
94  * <li><code>d < 0</code>: Underruns! Decrease read frequency =>
95  * Increase cycle time by a fixed number of milliseconds.</li>
96  * <li><code>d > 0</code>: Overruns! Increase read frequency =>
97  * Decrease cycle time by a fixed number of milliseconds.</li>
98  * </ul>
99  * </li>
100  * </ul>
101  * <p><br/></p>
102  *
103  * If the contention counter has a steep rise then both, the producer and consumer are
104  * too aggressively trying to set and read data. As a mitigation strategy the party with the
105  * lower priority must be forced to yield whilst the higher priority party has the critical section.
106  * You can use the {@link #IsLocked()} method for that:
107  * <p><br/></p>
108  *
109  * <pre>
110  * while (myBuffer.IsLocked())
111  * {
112  * try
113  * {
114  * Thread.sleep (2);
115  * } catch (ThatInterruptedExceptionISeeNoUseFor e){}
116  * }
117  * </pre>
118  * <p><br/></p>
119  *
120  * This is not a pretty solution, but it works - best use sparingly, if you have to.
121  * <p><br/></p>
122  *
123  * Don't assume the counters are precise. Overall access to the counters isn't thread safe
124  * (it seemed overkill to implement that, too). Whilst each counter is using an atomic
125  * integer (and should be thread safe by itself), we still can have scenarios such as one
126  * thread incrementing the underrun counter whilst another thread reads the overrun counter.
127  * Due to lack of overall counter thread safety the counters can only serve as guideline
128  * to assess the various reasons for dropped data. Them counters aren't brilliant, but good
129  * enough for simple diagnosis.
130  * <p><br/></p>
131  *
132  * Clients can customize what's returned to the consumer thread when no valid data is
133  * available upon call to {@link #Get()}. Default is to return an empty sample chunk
134  * (in effect saying - zero samples available). But it means that for every invalid
135  * {@link #Get()} call we create a new empty {@link FloatBuffer} which doesn't really
136  * feel that good. Therefore there's also the possibility to have {@link #Get()}
137  * return <code>null</code> which allows consumers to operate in a similar fashion
138  * as it's done with {@link AtomicInteger#compareAndSet(int, int)}:
139  * <p><br/></p>
140  *
141  * <pre>
142  * // At some place:
143  * TAtomicBuffer myBuf = new TAtomicBuffer (ECopyPolicy.kCopyOnGet, EIfInvalidPolicy.kReturnNull)
144  *
145  * // Inside consumer thread
146  * do
147  * {
148  * chunk = myBuf.Get ();
149  * try {
150  * Thread.sleep (yieldTime); // Can be a few mSec so the processor doesn't grind to a halt.
151  * } catch (ThatInterruptedExceptionISeeNoUseFor e){}
152  * } while (chunk == null);
153  *
154  * // Now do nifty stuff with the sample chunk.
155  * </pre>
156  * <p><br/></p>
157  *
158  * For synchronization between producer and consumer the <code>TAtomicBuffer</code>
159  * makes use of an {@link AtomicInteger} flag; more specifically, the
160  * {@link AtomicInteger#compareAndSet(int, int)} method. This method is guaranteed
161  * to be atomic, i.e. allow access to the enclosed integer only by a single thread at
162  * a time without blocking the other threads. We don't use the standard
163  * <code>synchronized</code> mutex as that one does block threads which haven't
164  * acquired it. This can block the time critical thread, leading to priority inversion
165  * (where the low priority thread blocks the high priority one). Additionally,
166  * on some platforms the <code>synchronized</code> primitive is quite costly, i.e.
167  * time consuming. The {@link AtomicInteger#compareAndSet(int, int)} is much faster.
168  *
169  * @author Peter Hoppe
170  */

255  /**
256  * Returns the newest sample data. Meant for the consumer thread.
257  * Note that at some stage there's a deep copying;
258  * therefore we will never return the original {@link FloatBuffer} object.
259  * <p><br/></p>
260  *
261  * If there was no new sample data (i.e. stale data = that particular chunk has
262  * already been collected before) or if there was contention with the producer
263  * thread then we'll return either an empty sample chunk or null. See
264  * {@link TAtomicBuffer introduction} for further information.
265  *
266  * @return The newest sample data, or empty chunk, resp. <code>null</code>.
267  */
268  public FloatBuffer Get ()
269  {
270  boolean isLocked;
271  FloatBuffer ret;
272 
273  if (fCopyPolicy == EAtomicBuffer_CopyPolicy.kCopyOnGet)
274  { /* Consumer has low priority, i.e. can wait until the lock is free. */
275  _Lock ();
276  ret = _DataGet (true);
277  _Unlock ();
278  }
279  else if (fCopyPolicy == EAtomicBuffer_CopyPolicy.kCopyOnSet)
280  { /* Consumer has high priority, i.e. will give up trying if it can't acquire the lock */
281  isLocked = _TryLock (); /* [110] */
282  if (isLocked)
283  {
284  ret = _DataGet (false);
285  _Unlock ();
286  }
287  else
288  { /* Consumer clashed with producer. Increment
289  contention counter and return empty chunk
290  or null. */
291  fStats.IncrementContentions ();
292  if (fIfInvalidPolicy == EAtomicBuffer_IfInvalidPolicy.kReturnEmpty)
293  {
294  ret = FloatBuffer.allocate (0);
295  }
296  else
297  {
298  ret = null;
299  }
300  }
301  }
302  else
303  {
304  /* No copying. This is dangerous, but sometimes necessary.
305  * Clients must make sure the producer or the consumer do
306  * their own data copying. Since we can't tell who's
307  * high and who's low priority we treat this case as if both,
308  * producer and consumer are high priority. */
309  isLocked = _TryLock ();
310  if (isLocked)
311  {
312  ret = _DataGet (false);
313  _Unlock ();
314  }
315  else
316  { /* Consumer clashed with producer. Increment
317  contention counter and return empty chunk
318  or null. */
319  fStats.IncrementContentions ();
320  if (fIfInvalidPolicy == EAtomicBuffer_IfInvalidPolicy.kReturnEmpty)
321  {
322  ret = FloatBuffer.allocate (0);
323  }
324  else
325  {
326  ret = null;
327  }
328  }
329  }
330 
331  return ret;
332  }

428  /**
429  * An internal work method: Retrieves (a deep copy of) the newest sample chunk,
430  * or empty chunk, resp. <code>null</code> if there's stale data. If there was
431  * just stale data we will increment the underrun counter.
432  *
433  * @param doCopy Return a deep copy, not the original FloatBuffer object.
434  * @return The (deep copy of the) latest sample chunk.
435  */
436  private FloatBuffer _DataGet (boolean doCopy)
437  {
438  FloatBuffer ret;
439 
440  /*
441  * Note, it's mandatory to NEVER do: fBuffer=FloatBuffer.allocate (0)!
442  * Otherwise, if doCopy == false, we will set ret=fBuffer,
443  * then fBuffer=FloatBuffer.allocate(0) which is in effect
444  * ret=FloatBuffer.allocate (0) since ret==fBuffer! Thus the consumer
445  * will receive an empty data set even though there was valid data!
446  * However we neither want to pass on stale data to the consumer.
447  * Our compromise is: When data is stale, we create an empty data
448  * set or null and pass that back to the consumer.
449  */
450  if (! fHasBeenCollected)
451  { /* Ah! Some fresh data is ready to collect! */
452  if (doCopy)
453  { /* The consumer thread is low priority. */
454  ret = _GetDeepCopy (fBuffer); /* [100] */
455  }
456  else
457  { /* The consumer is high priority - can't afford to copy. */
458  ret = fBuffer;
459  }
460  }
461  else
462  { /* Ouch! We are sitting on stale data! Increment underrun
463  counter and return empty chunk or null. */
464  fStats.IncrementUnderruns ();
465  if (fIfInvalidPolicy == EAtomicBuffer_IfInvalidPolicy.kReturnEmpty)
466  {
467  ret = FloatBuffer.allocate (0);
468  }
469  else
470  {
471  ret = null;
472  }
473  }
474 
475  /* Mark data as stale. For simplicity we ignore whatever fHasBeenCollected was before */
476  fHasBeenCollected = true;
477 
478  return ret;
479  }
480 
481  /**
482  * Store (a copy of the) new sample chunk, whether the consumer collected or not.
483  * If there was stale data (i.e. uncollected since last _DataSet(..)
484  * we will increment the overrun counter.
485  *
486  * @param fb The sample chunk to be stored.
487  * @param doCopy If <code>true</code>, store a deep copy instead of
488  * the original.
489  */
490  private void _DataSet (FloatBuffer fb, boolean doCopy)
491  {
492  if (doCopy)
493  { /* The producer thread is low priority. */
494  fBuffer = _GetDeepCopy (fb);
495  }
496  else
497  { /* The producer is high priority - can't afford to copy. */
498  fBuffer = fb;
499  }
500 
501  if (! fHasBeenCollected)
502  { /* Ouch! We were sitting on stale data which means the consumer
503  wasn't able to collect in time. Increment the overrun counter */
504  fStats.IncrementOverruns ();
505  }
506 
507  /* Mark data as fresh. For simplicity we ignore whatever fHasBeenCollected was before */
508  fHasBeenCollected = false;
509  }

511  /**
512  * Creates a deep copy of the passed float buffer (for copy-on-write semantics).
513  *
514  * @param fb The {@link FloatBuffer} to be copied.
515  * @return The deep copy. Essentially, this is a new (separate) object.
516  */
517  private FloatBuffer _GetDeepCopy (FloatBuffer fb)
518  {
519  int i;
520  int nNums;
521  float v;
522  FloatBuffer ret;
523 
524  nNums = fb.capacity ();
525  ret = FloatBuffer.allocate (nNums);
526  if (nNums >= 1)
527  {
528  for (i = 0; i < nNums; i++)
529  {
530  v = fb.get (i);
531  ret.put (v);
532  }
533  }
534 
535  return ret;
536  }

554  /**

560  boolean isLocked;

564  isLocked = fFlag.compareAndSet (gkUnlocked, gkLocked);
565  } while (! isLocked); /* [120] */

578 
579  return ret;

591  {
592  isUnlocked = fFlag.compareAndSet (gkLocked, gkUnlocked);

598 /*
599 [100] It's safer to make a copy, as the producer thread
600  might overwrite values whilst the consumer still works
601  with the values retrieved.
602 
603 [110] We only lock if the consumer thread isn't retrieving values -
604  The deep copy operation in Get() takes more time than the
605  reference copy in the Set() method.
606  That way at least the producer thread doesn't have to wait
607  on what's effectively a spin lock (whilst we deep copy).
608  It's justifiable on the grounds
609  that we can simply discard updates to the buffer whilst the buffer
610  is being read by the consumer. This policy is especially effective when
611  the producer thread is a high priority thread and the consumer
612  thread is lower priority (e.g. audio core loop produces
613  sample chunks, GUI thread consumes the data. With this policy
614  we minimize wait time for the audio core thread. We simply say
615  that the GUI doesn't need fresh frames whilst it's updating itself.
616 
617 [120] This is, effectively, a spin lock.
618 
619 [130] In a previous version I did a silly mistake by writing
620 
621  else
622  {
623  fNumUnderruns.incrementAndGet ();
624  fBuffer = FloatBuffer.allocate (0); // silly mistake
625  }
626 
627  This must have been because I thoughtlessly duplicated the idea
628  from the Get() method
629 
630  else
631  {
632  fNumUnderruns.incrementAndGet ();
633  ret = FloatBuffer.allocate (0);
634  }
635 
636  which has the aim of returning something definite to the caller.
637  Similarly, in the Set() method I intended to bring fBuffer onto
638  something definite if the caller fails to acquire the lock.
639  This, of course, is faulty thinking - the last thing we want is to
640  clear the fBuffer whilst another thread is accessing it! With one
641  thoughtless line I would have killed the protection. I really wasn't
642  thinking whilst typing! I'm thankful I discovered it.
643 */
