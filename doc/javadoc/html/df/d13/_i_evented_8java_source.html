21 /**
22  * Base interface for the event system. Normally, we'd provide a dedicated
23  * class each type of event. This would scale well;
24  * each event sending class could simply define its own event classes,
25  * and we could put whatever features we desire into each event type.
26  * At runtime, upon firing an event we create an object of the desired
27  * event type and <code>post()</code> it with the global {@link TController}.
28  * Once the event has been handled we throw the event object away.
29  * This means at runtime we create a lot of throw-away objects which seems
30  * to be a stupid design to me (especially as some of the modules fire
31  * events at high frequency, such as {@link TTimer}).
32  * Therefore we simplify dramatically and assign an integer constant to each
33  * event type. With each event fired we post the (integer) event ID and the unique ID
34  * of the construct firing the event. This saves a lot of object creation and
35  * is a lot faster, too!
36  * Disadvantage is that we can't provide in-band data with a simple call to
37  * {@link #OnEvent(int)}. That's why we provide additional methods that allow
38  * for arguments with each event (e.g. {@link #OnEvent(int, int)}.
39  *
40  * @author Peter Hoppe
41  */
