170  /**
171  * A spin lock, acquiring a critical section.
172  */
173  private void _Lock ()
174  {
175  boolean isLocked;
176 
177  do /* [100] */
178  {
179  isLocked = fLock.compareAndSet (gkUnlocked, gkLocked);
180  } while (! isLocked);
181  }
182 
183  /**
184  * A spin lock, releasing a critical section.
185  */
186  private void _Unlock ()
187  {
188  boolean isUnlocked; /* [100] [110] */
189 
190  do
191  {
192  isUnlocked = fLock.compareAndSet (gkLocked, gkUnlocked);
193  } while (! isUnlocked);
194  }
195 }

197 /*
198 [100] Using a simple spin lock for synchronization: Since it's a loop it smells
199  of program freeze. However, in practice the loop should be finished within
200  one or a few cycles, as the critical sections follow a very straight forward,
201  linear code execution flow - in effect, just a few statements (function call
202  with everything that is run whilst executing it) per critical
203  section. Suppose:
204  * Thread A calls Get(0)
205  * Thread A calls _Lock()
206  * Thread A enters the spin lock and calls compareAndSet().
207  * Thread A acquires the lock. Critical section now protected.
208  * Just at that time, Thread B calls Get(1)
209  * Thread B calls _Lock(). However, fLock is already set to locked, so
210  Thread B sits in the spin lock. It shouldn't take long (e.g. a few cycles).
211  * Object is being retrieved from the TArrayMap. This is what Thread B is waiting for.
212  * Thread A calls _Unlock()
213  * Thread A enters the spin lock in _Unlock().
214  * Thread A calls compareAndSet which sets the state to unlocked.
215  * Thread B reaches the compareAndSet statement and locks the thread.
216  In effect, Thread B has to wait a very short time - whilst the object is being retrieved.
217  Therefore, in practice, the spin lock won't hold up Thread B for very long.
218 
219  I could use a simple synchronized primitive, but somewhere I read that
220  using atomic variables has less costs than using synchronized. Besides, I'd
221  like to port this program into C++ (QT toolkit) which I know to offer
222  atomic variables, but I don't know what replacement exists for Java's
223  synchronized primitive.
224 
225 [110] I don't know whether I actually need a spin lock here or whether a single
226  compareAndSet statement suffices (without loop). For by the time execution
227  reaches the compareAndSet statement all other threads are already excluded.
228  It still feels better - the AtomicInteger itself can be accessed by multiple
229  threads simultaneously (e.g. the exclusive thread calls _Unlock() whilst
230  another thread calls _Lock() at all times. I don't know whether a single
231  compareAndSet statement without loop is enough - just to be safe I use the same
232  spin lock mechanism as in _Lock().
233 */
